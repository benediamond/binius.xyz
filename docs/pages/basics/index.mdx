# Basics [An introduction to Binius]

_Binius_ is a state-of-the-art noninteractive succinct proof system. It comprises a theoretical cryptographic protocol (expounded in a sequence of recent technical manuscripts [[DP23](https://eprint.iacr.org/2023/1784)] [[DP24](https://eprint.iacr.org/2024/504)]), a production-grade [Rust implementation](https://gitlab.com/IrreducibleOSS/binius) of that protocol, and finally a documentation suite, featured on this very website.

Binius's underlying cryptographic protocol operates natively over _towers of binary fields_, mathematical objects which, on the one hand, efficiently arithmetize natural problems of interest (like standard hash functions), and, on the other, admit extremely efficient implementations in both software and hardware. Binius's full codebase resides on GitLab at [`IrreducibleOSS / binius`](https://gitlab.com/IrreducibleOSS/binius/). That codebase contains not just Binius's core prover and verifier logic, but moreover various examples which serve to exhibit that system's end-to-end behavior on tasks of essential interest (like Merkle–Patricia trie inclusion proving).

In this area of the site, we collect basic resources, designed to jump-start those planning to build on Binius. Here, we mainly focus on Binius's API and on the [arithmetization](/basics/arithmetization) scheme we use to capture real-life statements and problems. We defer Binius's cryptographic inner-workings to our [blueprint](/blueprint) section.

[**Binius's Arithmetization.**](arithmetization) In this section, we define abstract tools and objects which let us efficiently capture _meaningful_ real-life statements and problems.

Binius introduces a new arithmetization scheme, called _M3_ (short for _multi-multiset matching_). M3 features a plurality of _tables_, to each of which arbitrary polynomial _gate constraints_—operating row-wise—may be imposed. We moreover link and connect these tables using a new abstraction we call _channels_. Channels are abstract objects, _to_ and _from_ which tuples of data may be "pushed" and "pulled". Our arithmetization paradigm encompasses _not just_ a plurality of tables (each, again, bearing gate constraints), but moreover so-called _flushing rules_, which dictate how elements from these tables should be pushed to and pulled from channels. A channel is said to be _balanced_ when the pushes made to it exactly match the pulls made from it.

As it turns out, many essential problems, like [Merkle–Patricia trie inclusion](/basics/arithmetization/mpt), can be captured as M3 instances. For more on arithmetization, start [here](/basics/arithmetization).
