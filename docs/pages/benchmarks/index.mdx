## Notes and Explanations

All performance numbers were collected on a GCP `c3-standard-44` machine, with 22 cores of Intel _Sapphire Rapids_ type. All binaries were compiled and run in release mode, with the environment variable `RUSTFLAGS="-C target-cpu=native"`. All Plonky3 examples were also run in parallel—i.e., with the `--features parallel` flag. We note that the proving time includes all witness and trace generation as well.

Our benchmarks above capture hashing-related tasks, themselves involving the execution of many internal _permutation functions_. We arranged things in each case so as to hash at least 1 MB of data. We ended up overshooting this slightly above, since we need tables of power-of-two height (each benchmark actually hashes somewhere between 1.049 and 1.1 MB of data).

For u32-add and other binary operations we are doing slightly over 1 million operations per SNARK. 

We explain a bit more thoroughly how we derive "total data hashed" as a function of how many permutations we execute. For each $\text{Keccak-}f$ permutation, we count only those semantically meaningful bytes directly absorbed into the sponge's _rate_ portion. That is, only 136 semantically meaningful bytes get absorbed per $\text{Keccak-}f$ permutation (even though the total state size upon which that permutation acts is 200 bytes). In simple terms, we count as "hashed bytes" the number of permutations times 136, not the number of permutations times 200. In the case of Grøstl, we've actually only implemented that hash function's $P$ permutation thus far. To process a 64-byte block of input data, that hash function must apply $P$ exactly once and $Q$ exactly once. We thus fudge slightly, by acting as if we were doing both $P$s and $Q$s, even though we are actually doing doubly many $P$s. Thus, we count 64 bytes "hashed" for every _two_ executions of the $P$ permutation. In each $\text{Keccak-}f$ benchmark, we run $2^{13}$ permutations; in each of the others, we run $2^{14}$.
