# The Refinement Polynomial [A virtual operation that "splits" a committed large-field polynomial]

In this page, we discuss _refinement_ and _packing_, two dual notions that sit at the center of ring-switching. Indeed, as we've said, we've thus far only [committed](/blueprint/cryptography/commitment) large-field polynomials. On the other hand, given any large-field polynomial, we can _freely decide to view it_ as a small-field polynomial, merely by refining the boundary points which divide adjacent field elements. The point is to make this formal data-casting operation _algebraic_, so that we can treat it using the ring-switching reduction.

## The Refinement Polynomial

Essentially, $\mathcal{T}_\tau$-elements concretely look like length-$2^\tau$ strings of bits. We're going to pick a _refinement_ factor $\kappa \in \{0, \ldots , \tau\}$. The basic operation is this: if I _start_ with a length-$2^\ell$ list containing $2^\tau$-bit strings, then, by "chopping them up" into $2^\kappa$ substrings a piece, I can cook up a length-$2^{\ell + \kappa}$ list of $2^{\tau - \kappa}$-bit strings.

:::info
For example, let's take $\tau = 7$, $\ell = 20$, and $\kappa = 4$. Thus, I'm given as input a huge list, of length $2^{20}$, _each_ of whose elements is a 128-bit string.

I'm going to chop up each of these 128-bit strings into $2^4 = 16$ substrings. Each of these resulting substrings will be of length $2^{7 - 4} = 8$ bits.

The total number of _8-bit_ strings I'm left with, after this process concludes, will be $2^{20 + 4} = 2^{24}$.
:::

This is a procedure we just introduced on lists. On the other hand, using the [dictionary](/blueprint/primer) between lists and multilinears, we can always equally express the procedure in terms of _multilinears_, too. Thus our input is a multilinear $t(X_0, \ldots , X_{\ell - 1})$ over the huge field $\mathcal{T}_\tau$. By performing the above procedure on $t$'s vector of Lagrange coefficients, we get a _further_ vector. Upon interpreting _that_ vector as the vector of Lagrange coefficients of some further polynomial, we obtain a polynomial $t'(X_0, \ldots, X_{\ell + \kappa - 1})$ over $\mathcal{T}_{\tau - \kappa}$. We call $t'(X_0, \ldots, X_{\ell + \kappa - 1})$ the _refinement virtual polynomial_.

## The Packed Polynomial

We can always reverse this process, in a fairly straightforward way. Indeed, if we _start_ with a polynomial $t(X_0, \ldots, X_{\ell - 1})$ over $\mathcal{T}_\iota$, let's say, we can choose to _pack_ its coefficients. In this side of things, we arrange $t(X_0, \ldots, X_{\ell - 1})$'s vector of Lagrange coefficients into _chunks_, and concatenate each chunk. In this way, we obtain a polynomial with _fewer_ coefficients, but over a _larger_ field. More precisely, if our chunk size is $2^\kappa$, then the end result will be $t'(X_0, \ldots , X_{\ell - \kappa - 1})$, a polynomial over $\mathcal{T}_{\iota + \kappa}$. The number of _variables_ has gotten smaller, by $\kappa$, while the tower height has increased by $\kappa$.

## Binary Towers

The key fact that underlies everything that comes next is that these packing and refinement operations are _algebraically meaningful_. If we were working in the prime-field setting, it simply would _not_ be algebraically meaningful to slice and dice the _bit-representations_ of field elements in the way that we're doing now. Nor, for that matter, would these operations be meaningful if we used _classical_ binary fields (defined as quotients of univariate polynomial rings).

Binius, uses _towers of binary fields_, binary fields of a special sort. The math here is a bit involved (though very natural); we defer as much as possible to [[DP23](https://eprint.iacr.org/2023/1784.pdf), § 2.3]. For now, we just record without proof a basic fact that explains the algebraic meaningfulness of our refinement and packing operations.

Indeed, we fix two tower heights $\mathcal{T}_\iota \subset \mathcal{T}_\tau$. We write $\kappa \coloneqq \tau - \iota$. As is true for _any_ field extension, we can always view $\mathcal{T}_\tau$ as a vector space over its own subfield $\mathcal{T}_\iota$. In [[DP23](https://eprint.iacr.org/2023/1784.pdf), § 2.3], we describe a _particular_ basis $(\beta_0, \ldots , \beta_{2^\kappa - 1})$ of $\mathcal{T}_\tau$ _over_ $\mathcal{T}_\iota$, called the _the multilinear monomial basis_.

Above, we talked field elements as if they were _just_ strings of bits, and also about "splitting" and "concatenating" these strings. Of course, field elements are _not just_ strings of bits. In fact, they only come to be representable in this way at all once we choose a basis of our field over its subfield $\mathbb{F}_2$ (the "bitstrings" end up being the _coordinates_ our various elements with respect to this basis).

Splitting and concatenating aren't algebraic operations either, but _basis decomposition_ is. If we start with a big-field element $\alpha \in \mathcal{T}_\tau$, then, definition of the basis $\beta_0, \ldots , \beta_{2^\kappa - 1}$, we get a unique sequence of $\mathcal{T}_\iota$-valued _coordinates_ $\alpha_0, \ldots , \alpha_{2^\kappa - 1}$ for which $\alpha = \sum_{i = 0}^{2^\kappa - 1} \alpha_i \cdot \beta_i$ holds. Thus, the relationship between $\alpha$ and $\alpha_0, \ldots , \alpha_{2^\kappa - 1}$ is a basis-decomposition, or an _expression in coordinates_.

Above, while we were superficially talking about splitting and concatenating, we were actually talking about basis-decomposition and basis-combination. This is because of a key "commutativity" property that towers exhibit: the chain of subfields $\mathbb{F}_2 \subset \mathcal{T}_\iota \subset \mathcal{T}_\tau$ (and associated bases) "plays nice", in a certain sense. That is, if we express a $\mathcal{T}_\tau$-element as a list of $\mathcal{T}_\iota$-coordinates, and then splinter _those_ into $\mathbb{F}_2$-coordinates (i.e. bits), then the effect is the same as if we directly expressed the initial $\mathcal{T}_\tau$-element in $\mathbb{F}_2$-coordinates.

This "scale-invariant basis" property is described further in [[DP23](https://eprint.iacr.org/2023/1784.pdf), § 2.3].
