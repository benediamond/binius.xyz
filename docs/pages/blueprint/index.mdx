# Blueprint [Cryptographic details about our backend]

In this site section, we explain the cryptographic backend of Binius64.

## Site Directory

We've split this site area into a few subsections.

<Callout type="note">
<Collapsible>
[**Math Background.**](math) This section contains a crash course on the mathematical objects we use.

Start here if you're new. We can't promise to cover everything, but it'll be a useful starting point.
</Collapsible>
</Callout>

<Callout type="note">
<Collapsible>
[**Constraint Systems.**](constraints) _Constraint systems_ are how we specify the computations we set about proving and verifying. Think somewhere in between "circuits" and "R1CS".

Binius64 introduces a new constraint system. Our constraint system basically takes R1CS, makes it operate fundamentally on 64-bit unsigned integers, and moreover enriches it by natively supporting constraints that act upon (logical, arithmetic, etc.) _bitshifts_ of this witness data. We also bake in support for unsigned, 64-bit integer multiplication.

The constraint system we get out is predictable and simple, but much more useful and efficient than plain, vanilla R1CS in characteristic 2.
</Collapsible>
</Callout>

<Callout type="note">
<Collapsible>
[**AND Constraints.**](ands) To check whether the prover's data satisfies the constraint system's AND constraints, we need to do a few steps.

We first arithmetize the problem of AND-constraint satisfaction, by representing that problem as one about [multilinear polynomials](math/multilinears). We actually use [oblong](ands/oblong) multilinears, an idea that comes from Gruen [[Gru24](https://eprint.iacr.org/2024/108), § 5.1]'s _univariate skip_. That technique ends up working especially nicely for us, since we natively use the _64-bit word_ dataype. After running the univariate skip, we wind up with a complicated, Spartan-like sumcheck, called the [shift reduction](ands/reduction), which we also treat in detail.
</Collapsible>
</Callout>

<Callout type="note">
<Collapsible>
[**MUL Constraints.**](multiplication) To check whether the prover's data satisfies the constraint system's MUL constraints, we use a totally different technique, this time based on GKR.

Because exponentiating a fixed base by the product of two values is equivalent to exponentiating that base by the two values separately in succession, we can express multiplication in terms of exponentiation. This lets us work "in the exponent", and make use of a GKR-based trick.
</Collapsible>
</Callout>

<Callout type="note">
<Collapsible>
[**Polynomial Commitment.**](commitment) The "lowermost" layer of our cryptographic stack deals with the assessment of evaluation claims about multilinear polynomials.

We don't say a ton here. Mostly, we refer out to [[DP24](https://eprint.iacr.org/2024/504)], which describes in detail the polynomial commitment scheme that we use.
</Collapsible>
</Callout>
