# The Collatz function and iterating functions with branching [A toy function and some generalizations]
We explain how to use M3 to first instantiate a well-known problem in classical number theory and then generalize it. For the sake of simplicity, we will work with natural numbers, though everything may be ported over to tower fields with a bit of care (and some bounds).
## Collatz orbits
Consider the following function $f$: on input $x$, where $x\in \{1,\ldots,\}$, 
* if $x$ is even, $f(x) = \frac{x}{2}$; and
* if $x$ is odd, $f(x) = 3x+1$.

The [Collatz conjecture](https://en.wikipedia.org/wiki/Collatz_conjecture) states that for every $x$, if I repeatedly apply $f$, I will eventually get to the number $1$. In particular, it would imply that the set $\{x, f(x), f(f(x)), f(f(f(x))),\ldots \}$ of iterated applications of $f$ on $x$ is *finite*. All evidence points to the answer being "yes", but a formal proof is still lacking. One way to think about why the conjecture is so difficult is that the pattern of "which branch do I take" is somewhat stochastic.

Let's say that a natural number $y$ is in the *Collatz orbit* of $x$ if $y=f(f(\cdots (f(x))))$, i.e., if after applying $f$ a bunch of times to $x$, we get $y$. We'd like to formalize the statement "$y$ is in the Collatz orbit of $x$" in M3. We encourage the reader to stop reading and try this themselves!

### Channels
We have a unique channel, of type signature $\mathbb N$.

### Tables
We will have two tables, $T_{\text{even}}$ and $T_{\text{odd}}$. 

The first table, $T_{\text{even}}$, has type signature $(\mathbb N, \mathbb N)$, while the table $T_{\text{odd}}$ has type signature $(\mathbb N, \mathbb N, \mathbb N)$. That is, $T_{\text{even}}$ has width 2 while $T_{\text{odd}}$ has width 3, and all of the cells in question have type "natural numbers".

#### Table constraints
The even table has a unique constraint:
* if $(a, b)$ is a row of $T_{\text{even}}$, then $2b = a$. In other words, the constraint witnesses that $a$ is even. Note that this witness is also the value $b=f(a)$ in this case.

The odd table has two constraints:
* if $(a, b, c)$ is a row of $T_{\text{odd}}$, then $2b + 1 = a$ and $3a + 1 = c$. In other words, the constraint witnesses that $a$ is odd and that $c = f(a)$.

#### Flushing rules
We have the following flushing rules:
* if $(a, b)$ is a row of $T_{\text{even}}$, we pull $b$ and push $a$.
* if $(a, b, c)$ is a row of $T_{\text{odd}}$, we pull $c$ and push $a$.

In other words, we push $a$ and pull $f(a)$.
### Boundary constraints
Suppose we want to prove that $y$ is in the Collatz orbit of $x$. Then our boundary constraints will start with an $x$ in the pull column and a $y$ in the push column:
:::info
Boundary constraints/"initial condition":
| Pushes                 | Pulls                  |
| :---------------------:| :---------------------:|
| `y` | `x` |

:::
### The claim
The claim is that $y$ is in the Collatz orbit of $x$ if and only if the prover can fill in the tables $T_{\text{even}}$ and $T_{\text{odd}}$ in such a way that, after flushing, causes the channel to be balanced.

## Iterating functions with branching
Putting on our generalization hats, we see that the Collatz orbit problem is actually a special case of a more general type of problem: suppose we have some function $g$ that is defined in a _piecewise/branching_ way and we would like to prove/witness that $y$ is in the iterated-$g$-orbit of $x$ for some $x$ and $y$. The iterated behavior of even simple arithmetic branching functions can be extremely rich (c.f. [Unpredictable Iterations](https://gwern.net/doc/cs/computable/1972-conway.pdf) and [Fractran](https://en.wikipedia.org/wiki/FRACTRAN)).

We are interested in such situations because they highlight differences between M3 and other arithmetization/constraint satisfaction instances. In other arithmetization approaches, there is typically a _global execution trace_ which is responsible for orchestrating the order of operations of more primitive function calls (which might be delegated to other tables). In M3, each individual branch could have its own table computing/validing it; then it is _simply channel balancing_ that is responsible for the orchestration of all of the branches.
