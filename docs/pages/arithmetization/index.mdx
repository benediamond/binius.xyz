# Binius's Arithmetization [A new approach, based on the balancing of channels]
 
In SNARK design, _arithmetization_ refers to any among a number of techniques designed to reduce _computational problems_ to _mathematical problems_. Computational problems are the ones we care about in day-to-day life. For example:

:::note
Here's an Ethereum block. I claim that its state root is correct, in the sense that, after running all of its transactions against a global state trie captured by its _previous block hash_, I indeed obtained a post-transaction state root which matches that contained in the block.
:::

Mathematical problems are the ones amenable to treatment by SNARKs. These latter statements typically involve _finite fields_, _tables_ and _polynomials_. For example:
:::note
Here is a cryptographic commitment to a table of values, each an element of a finite field. I claim that this collection of multivariate polynomials vanishes identically over that table's rows.
:::

Cassical arithmetization schemes, like Starkware's _AIR_ (Algebraic Intermediate Representation) and Plonky3's _PLONKish_ scheme roughly take this form. Generally, we will also want some mechanism to link the table's adjacent rows, as well as _boundary constraints_, which serve to mandate that the table's top and bottom rows (say) take prescribed values.
