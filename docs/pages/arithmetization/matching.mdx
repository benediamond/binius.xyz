# M3: Multi-Multiset Matching
 
## Introduction
Before diving into the details, we informally describe the M3 model of computation.  We fix a finite field $\mathbb F_q$.
We also fix a multiplicative generator $\alpha\in \mathbb F_q^{\times}$.

### An informal description of M3
M3 is a constraint-system model of computation. A primary novelty, in comparison to other models of
computation, is the use of a kind of generalized multisets, where the multiplicity of elements may be negative.
We call such gadgets *channels*.

In M3 one has several *tables* and several *channels*. To each table $T$, there is a list of polynomial
constraints that each row must satisfy. However, the order of the rows **does not matter**.

The channels facilitate communication between the tables. Roughly speaking, for each pair $(T,C)$ of a 
table and a channel and for each filled-in row $R$ of $T$, there are *flushing rules* that add "to be proven" statements and removes "already shown" claims to the channel.
(If we think of a channel as a generalized multiset, removing an element could cause the "multiplicity" of the element to become
negative.)

There are boundary constraints/initial conditions on the channels. The analogy in imperative programming is "inputs" and "outputs". Here, we think
of boundary constraints as being "things we would like to prove" and "things we already know".

A *satisfying assignment* of an M3 instance is a filling-in of some rows of each table $T$ such that:
* for each table $T$ and each associated polynomial constraint $f_{T,i}$, each row $R$ of $T$ satifies $f_{T,i}$; and
* after flushing each row of each table, each channel $C$ is balanced, i.e., is the *empty generalized multiset*.

## Formal description
### Channels
A channel $C$ consists of
* a positive integer $r_C$; and
* a function $i_C\colon \mathbb F_q^{r_c}\rightarrow \mathbb Z$.

We call a channel $C$ *balanced* if the function $i_C$ is identically 0.

Psychologically
* we think of each channel as have an associated tuple of *types*; while these primitive
types may include: $32$-bit unsigned integers, op-codes, field elements themselves, etc., we will always represent them as field elements.
* a channel may be thought of as a generalized multiset whose elements are in $\mathbb F^{r_c}$ and whose multiplicities may be positive or negative.
The function $i_C$ is then the *indicator function* for this generalized multiset.
* an element occuring with multiplicity $1$ corresponds to "a claim that must be shown"; an element occuring with multiplicity $-1$
corresponds to "a claim that has already been shown". (Similarly, an element occuring with positive multiplicity $k$ corresponds to an element
that "must be shown $k$ times".)
* a balanced channel corresponds to the empty generalized multiset.

### Constraint Tables
A *constraint table* consists of:
* a table/grid, with with $n_T$ columns and a maximum of $m_T$ rows.
* a finite list $f_{T,i}$ of $n_T$-variate polynomials with coefficients in $\mathbb F_q$;
* for each channel $C$, there are
    * a collection of $r_C$-tuple of $n_T$-variate polynomials $(\text{push}_{T,C,j})_j$; and
    * a collection of $r_C$-tuple of $n_T$-variate polynomials $(\text{pull}_{T,C,j})_j$.

Note that each collection may be empty. The most common use case will be where, for any given $(T,C)$, the collection
will either be empty or consist of a single $r_C$-tuple of $n_T$-variate polynomials.

Psychologically
* we think of the elements of the table as witnessing some part of the execution trace of the a subroutine of our desired computation.
* the polynomial constraints $f_{T,i}$ on the rows ensure that these "primitive execution steps" are correct.
* for any $j$, we may think of $\text{push}_{T,C,j}$ as a machine that takes as input elements of $\mathbb F_q^{n_T}$ and outputs
elements of $\mathbb F_q^{r_C}$. (The same is naturally true of $\text{pull}_{T,C,j}$.)
* the pushing and pulling polynomials allow for coordination with the channels, as we will see in the description of flushing.

### Flushing
Once a row $R$ in a routing table $T$ has been filled, it may be *flushed*; this means that for
each channel $C$,
* for each $j$, the polynomial $\text{push}_{T,C,j}$ is applied to $R$ to obtain an element of $x_j:=\text{push}_{T,C, j}(R)\in\mathbb F_q^{r_C}$. We then
increment $i_C(x_j)$, i.e., do `i_C(x_j) += 1`.
* for each $j$, the polynomial $\text{pull}_{T,C, j}$ is applied to $R$ to obtain an element of $y_j:=\text{pull}_{T,C,j}(R)\in\mathbb F_q^{r_C}$. We then
decrement $i_C(y_j)$, i.e., do `i_C(y_j) -= 1`.

Psychologically
* for a given row $R$ of a routing table $T$ and a given channel $C$, the flushing rules generate
what we loosely referred to as "what has been proven" ($y$) and "what remains to be proven" ($x$) claims.
    * if we imagine the channel as consisting of claims needing to be shown, then pulling and pushing "replaces" a claim that needs to
    be shown with a (presumbly slightly easier) claim that needs to be shown.
* flushing rules permit *communication/coordination* between routing tables.
### Boundary constraints
At the start of an M3 instance, the channels will usually not be balanced, i.e., $i_C$ will not not the zero-function.
We call these initial conditions the *boundary constraints*. Boundary constraints may be either hard-coded
into the M3 problem or instance-specific.

Psychologically
* if we imagine channels as generalized multisets, then typically the boundary constraints correspond to: "what we want to show" (which occurs with multiplicity $1$
and "what we know" (which occurs with multiplicity $-1$)

### Model of computation
An M3 instance is specified by a set of routing tables $\{T\}$, a set of channels $\{C\}$, and a set of flushing rules. Note
that the channels may not start out balanced.

A *satisfying assignment* of the M3 instance with boundary constraints corresponds to filling in the
constraint tables with elements of $\mathbb F$ such that the following hold:
* for every contraint table $T$ and every polynomial constraint $f_{T,i}$ associated to $T$, each row of $T$ satisfies $f_{T,i}$; and
* after applying all of the flushing rules $\text{push}_{T,C}$ and $\text{pull}_{T,C}$, every channel is balanced.

It is easy to see that both the search and decision version of M3-satisfiability are NP-complete problems.

### The name
One convenient way of representing a channel $C$ is as a *pair of multisets* $S_C$ and $S'_C$, with elements in $\mathbb F_q^{r_C}$.
In particular, the multiplicities of the elements of $S_C$ and $S'_C$ are all positive. (However, the multisets are not necessarily
disjoint!) Without loss of generality, pushing only affects $S_C$ and pulling only affects $S'_C$. Then checking that a channel is balanced
will amount to checking that the multisets $S_C$ and $S'_C$ are the same.

Hence, checking the satisfiability of an M3 instance amounts to *matching multiple multisets*.

## Motivation
There are several interrelated motivations for M3. At this point, it is useful to compare to PLONKish arithmetization.
### PLONKish arithmetization

PLONKish arithmetization works as follows. One has a large table/grid $T$, of dimensions $m\times n$, and a list of $n$-variate polynomials $f_{i}$.
Moreover, there are traditionally also copy-constraints, which decree that given subsets of positions of the grid $T$ must all have the same value.
As an alternative to copy-constraints , one may impose shifted-column constraints, which force certain columns to be (cyclic or zero-padded) shifts of another column.
(Shifted-column constraints are especially practical for proof systems whose underlying polynomial commitment scheme is for
multilinear polynomials.)

A *satisfying assignment* of a PLONKish instance is an assignment of field elements to the grid slots $T$ ("filling in $T$") such that
for every row, every $n$-variate polynomial is satisfied and moreover the copy-constraints or shifted-column constraints are all satisfied.

In particular, the PLONKish model has the following two properties:
* there is one large table and each row must satisfy the *same polynomial constraints*; and
* the order of the rows matters due to copy-constraints or the shifted-column constraints.

### Overhead
In PLONKish systems, there are (at least) the following two types of overhead.
* *Uniformity Overhead*. It is common for the the constraint polynomials to be sparse. In other words,
for a given constraint polynomial, most columns are not involved.
* *Padding Overhead*. Typically, when one commits to the columns of a table, one pads to force the height
of the table to be a power of 2. (This is for concrete efficiency reasons having to do with the PCS).

Routing tables allow one to circumvent the former, and any approach with multiple tables has a tendency to shrink the required padding overhead).

### Non-temporal nature
In the model, note that the order of the filled-in rows in the routing tables is irrelevant. Moreover,
there simply *is no notion of order* for the channels. We think of this as an extreme form of non-determinism.
The ordering is instead replaced by the flushing rules and checking that the channels are balanced; one may think
of the flushing rules as a complex orchestration of the computation at hand.

### Fibonacci
As usual, we start with the example of computing the $n^{\text{th}}$ element of the Fibonacci sequence. 
It turns out to be mildly more convenient to prove both the values of $F_n$ and $F_{n+1}$ if $n$ is odd and $F_{n-1}$ if $n$ is even.
Here is one example for using the M3 constraint system to encode this problem.

We have one channel $C$, with $r_C=3$. The elements of $C$, $(z_0,z_1, z_2)$, have the following meaning: if $z_0 = \alpha^k$, then $F_k = z_1$
and $F_{k+1} = z_2$. (**TODO:** should I put the second thing also in the exponent or is `u32_add` easier?)

Our boundary constraints will then be: $(\alpha^{n-1}, F_{n-1}, F_n)$ with multiplcity $1$ and $(1, 1, 1)$ with multiplicity $-1$.

The constraint table $T$ has five columns. An element $(x_0,x_1,x_2,x_3,x_4)$ corresponds to the following statement. Suppose
$x_0 = g^i$. Then if  $F_{i-3} = x_1$ and $F_{i-2} = x_2$, then $F_{i-1} = x_3$ and $F_{i} = x_4$. In particular, there are two
polynomial constraints on $T$:
1. $ x_1 +_{\text{u32-add}} x_2 = x_3.$
2. $ x_2 +_{\text{u32-add}} x_3 = x_4.$

The flushing rules are as follows.
* $\text{push}_0(X_0,X_1,X_2,X_3, X_4) = (X_0*\alpha^{-3}, X_1)$
* $\text{push}_1(X_0,X_1,X_2,X_3, X_4) = (X_0*\alpha^{-2}, X_2)$
* $\text{pull}_0(X_0,X_1,X_2,X_3, X_4) = (X_0 * \alpha^{-1}, X_3)$
* $\text{pull}_1(X_0,X_1,X_2,X_3, X_4) = (X_0 , X_4)$

### Interactive oracle proof
Given an M3 instance with boundary constraints, a prover wishes to prove to a verifier that they
know a satisfying assignment. Here is a simple way to accomplish this.
1. The prover fills in the constraint tables $T$ and, for each column of each routing table, sends an oracle of the column.
2. The verifier must check:
    (a) that the rows in each routing table $T$ all satisfy $f_{T,i}$; and
    (b) after applying the flushing rules, each channel is balanced.

Checking (a) amounts to checking that the columns of $T$ themselves all satisfy $f_{T,i}$. In our implementation,
the oracle will be a commitment to a multilinear polynomial whose evaluations on a Boolean hypercube are the values of the column.
Therefore, we will be reduced to a *zerocheck*.

We may check (b) via a Grand Product Argument. In more detail: to check that $S_C$ and $S'_C$ are equal,
the verifier picks $r\in \mathbb F$ and and then the prover and verifier engage in a protocol to check that
$$\prod_{s\in S_C}(s-r) = \prod_{s'\in S'_C}(s'-r).$$
There are a variety of ways one could implement this, but they all require the verifier querying the column oracles
from the routing tables together with knowledge of the flushing rules.
