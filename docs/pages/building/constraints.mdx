# Constraint Systems [How we capture problem statements systematically]

At the center of our circuit programming API is the `ConstraintSystem`, which is shared between prover and verifier, and describes how the prover's witness data is constrained.

As a circuit programmer you won't need to access any of its fields directly, but there are still some things that are useful to know about it. It looks like this:

```rs
pub struct ConstraintSystem<F: TowerField> {
	pub oracles: MultilinearOracleSet<F>,
	pub table_constraints: Vec<ConstraintSet<F>>,
	pub non_zero_oracle_ids: Vec<OracleId>,
	pub flushes: Vec<Flush>,
	pub max_channel_id: ChannelId,
}
```

The `ConstraintSystem` currently constrains witness data in 4 ways:

- **Virtual oracles**: These define relationships between columns that the verifier also knows about. An example of this is `Shifted`, which describes one column as a shift of another one.
- **Polynomial constraints**: These are also known as zerocheck constraints, and asserts that certain polynomial compositions of witness columns are equal to zero everywhere _(on the hypercube)_.
- **Nonzero oracles**: A list of columns that are not allowed to contain any zeros.
- **Channel flushes**: Requires that all pushes/pulls to a channel must balance each other out. If the tuple (4, 5) is pushed to a channel, the tuple (4, 5) must also be pulled for the channel to be balanced.

---

There are 3 relevant functions you need to know about once you have a `ConstraintSystem`. `prove`, `verify` and `validate_witness`. The signatures look something like this after all the generic type arguments are hidden (for readability):

```rs
pub fn validate_witness(
	constraint_system: &ConstraintSystem,
	boundaries: &[Boundary],
	witness: &MultilinearExtensionIndex,
) -> Result<(), Error>;
```

- This is a developer tool to ensure that your witness generation logic agrees with your constraints.

```rs
pub fn prove(
	constraint_system: &ConstraintSystem,
	log_inv_rate: usize,
	security_bits: usize,
	witness: MultilinearExtensionIndex,
	domain_factory: DomainFactory,
	backend: &Backend,
) -> Result<Proof, Error>;
```

```rs
pub fn verify(
	constraint_system: &ConstraintSystem,
	log_inv_rate: usize,
	security_bits: usize,
	boundaries: Vec<Boundary>,
	proof: Proof,
) -> Result<(), Error>;
```
