# Introduction [Crash course on zero-knowledge circuits]


## Gates

Circuits are constructed from gates connected by wires. **Wires** hold 64-bit values - inputs, outputs, and intermediate results. **Gates** define operations between wires, generating constraints that must be satisfied.

Gates compile high-level operations into primitive constraints, similar to how processors implement complex instructions through boolean logic. For example, comparisons like `icmp_ult` examine the most significant bit after subtraction.

The CircuitBuilder API provides methods for creating gates. Assume you need AND during verification computation. Writing `let output = builder.band(a, b)`:
- `a` and `b` are input wires
- `output` is an output wire
- `band` is a gate that generates constraints (one AND constraint in this case, though gates may be quite complex and generate many constraints)

This single statement informs three phases:

**Circuit Building** (compile time)
```rust
let output = builder.band(a, b);  // Creates a new wire and constraint
```
Within a larger circuit, this might allocate wire #7 for `output` and generate constraint #3: `(a) & (b) ⊕ (output) = 0`. The circuit accumulates these constraints as it builds.

**Witness Generation** (proving time)
```rust
// Example: Prover fills witness vector
// witness[wire_a] = 0xFF00, witness[wire_b] = 0x1234
// Computes: witness[wire_7] = 0xFF00 & 0x1234 = 0x1200
```
The prover allocates a witness vector holding values for all wires. When computing the `band` operation, it stores `0x1200` at index 7 (the output wire's position). Constraints aren't checked yet - the prover fills the entire witness vector first.

**Constraint Verification** (verification time)
```rust
// Constraint checks: (0xFF00) & (0x1234) ⊕ (0x1200) = 0
```
Now the constraint is verified. The verifier evaluates the constraint using the witness values. If any constraint evaluates to non-zero, the proof is invalid.

## Concepts by Example

The following examples progressively illustrate fundamental concepts: using existing circuits, private witnesses, data types, non-deterministic operations, and low-cost constraint optimization.

## Example 0: Using Existing Circuits - SHA256 Preimage

Real applications combine basic constraints into complex cryptographic operations. Consider proving knowledge of a SHA256 preimage - a fundamental pattern in privacy protocols:

```rust
// 64-byte message (8 words)
let message: Vec<_> = (0..8).map(|_| builder.add_witness()).collect();

// Private message length
let message_len = builder.add_witness();

// SHA256 digest (4 words = 256 bits)
let expected_digest: [_; 4] = core::array::from_fn(|_| builder.add_inout());

let sha256 = Sha256::new(
    &builder,
    message_len,
    expected_digest,
    message.clone(),
);
```

The SHA256 circuit internally implements the complete algorithm through message scheduling, compression functions, and digest finalization.

During witness generation:
```rust
let mut w = circuit.new_witness_filler();

// Message length
w[message_len] = Word(64);

// Message with nonce
let mut message_bytes = vec![0u8; 64];
let test_data = b"test message data";
message_bytes[..test_data.len()].copy_from_slice(test_data);
// Add nonce to remaining bytes
let nonce = 0x123456789ABCDEF0u64.to_le_bytes();
let nonce_start = test_data.len();
message_bytes[nonce_start..nonce_start + 8].copy_from_slice(&nonce);

sha256.populate_message(&mut w, &message_bytes);

// Expected digest
let hash = StdSha256::digest(&message_bytes);
let mut digest_bytes = [0u8; 32];
digest_bytes.copy_from_slice(&hash);

sha256.populate_digest(&mut w, digest_bytes);
```

## Example 1: Basic Constraints and Private Witnesses

Consider proving knowledge of a private value that, when masked with AND, produces a known result:

```rust
let mask = builder.add_constant_64(0xFF00);
let private = builder.add_witness();
let result = builder.band(private, mask);
```

`add_witness()` creates private wires only the prover knows. For public input/output values shared with the verifier, use `add_inout()` instead.

This creates a constraint enforcing:
```
result = private & mask
```

Note that constraints encode equality through XOR: in binary fields, `a ⊕ b = 0` if and only if `a = b`. XOR functions as subtraction, yielding zero for equal values. Therefore the actual constraint becomes:
```
(private) & (mask) ⊕ (result) = 0
```

During witness generation, the prover supplies `private = 0x1234` and computes `result = 0x1234 & 0xFF00 = 0x1200`. The constraint verification evaluates: `(0x1234) & (0xFF00) ⊕ (0x1200) = 0x1200 ⊕ 0x1200 = 0`.

The verifier confirms that some private value satisfying the constraint exists without learning its value.

## Example 2: Word Semantics

Binius64 represents all values as 64-bit words, encoding different data types through gate semantics:

- **Unsigned integers** (0 to 2^64-1): Operations such as `iadd`, `imul`, `icmp_ult`
- **Binary field elements** (GF(2^64)): Operations such as `bxor` for field addition
- **Booleans**: MSB-based convention for conditional logic
- **Bit patterns**: Raw 64-bit values with operations such as `band`, `bor`, shifts, rotations

```rust
let a = builder.add_constant_64(0xFF00FF00);
let b = builder.add_constant_64(0x00FF00FF);
let x = builder.add_constant_64(100);
let y = builder.add_constant_64(200);
let p = builder.add_constant_64(50);
let q = builder.add_constant_64(100);
let v = builder.add_constant_64(0xABCDEF12);
let mask = builder.add_constant_64(0xFFFF0000);

let xor_result = builder.bxor(a, b);
let zero = builder.add_constant_64(0);
let (int_sum, _) = builder.iadd_cin_cout(x, y, zero);
let cmp_result = builder.icmp_ult(p, q);
let bit_pattern = builder.band(v, mask);
```

**Boolean Representation**: Comparisons like `icmp_eq` and `icmp_ult` use the MSB (Most Significant Bit) to encode boolean values. The MSB conveniently happens to be the sign bit In two's complement. When MSB = 1, the condition is true; when MSB = 0, it's false. The lower 63 bits are undefined. This MSB-bool convention enables branchless conditional operations - the `select` gate uses the MSB to choose between two values without runtime branching.

While all values are 64-bit words at the constraint level, gates encode type semantics - `iadd` treats inputs as unsigned integers with carry chains, `bxor` performs field addition, comparison operations produce MSB-bool results.

## Example 3: Non-Deterministic Auxiliary Values

Consider modular arithmetic verification. The operation `a = b mod c` requires proving the existence of a quotient `q` such that `a = c × q + b`. The hint system handles `q`: the prover computes it outside the circuit and supplies it as a witness. The circuit verifies only that the relationship holds.

Such witnesses are **non-deterministic** - not determined by the circuit structure alone. While modulo has exactly one valid `q`, the quotient is an auxiliary witness - necessary for constraint satisfaction but not part of the statement being proven.

To verify `100 mod 83 = 17`:

```rust
// BigUint division for multi-limb numbers
let dividend = vec![builder.add_constant_64(100)];
let divisor = vec![builder.add_constant_64(83)];

// Hint computes quotient and remainder externally
let (quotient, remainder) = builder.biguint_divide_hint(&dividend, &divisor);
```

The hint creates auxiliary witness wires without generating constraints. The prover computes `100 ÷ 83 = 1 remainder 17` and fills these wires.

Constraints verify the relationship:

```rust
// Verify: dividend = divisor × quotient + remainder
let (hi, lo) = builder.imul(divisor[0], quotient[0]);
let sum = builder.iadd_cin_cout(lo, remainder[0], zero_wire).0;
builder.assert_eq("modulo_check", sum, dividend[0]);
```

## Example 4: Constraint Optimization - XOR Packing

Consider computing the XOR of multiple values. The naive approach chains individual XOR gates:

```rust
// Naive: 3 separate XOR gates, 3 constraints
let temp1 = builder.bxor(a, b);        // Constraint 1
let temp2 = builder.bxor(temp1, c);    // Constraint 2
let result = builder.bxor(temp2, d);   // Constraint 3
```

Each `bxor` generates a constraint:
```
(a) & (0xFF..FF) ⊕ (b) & (0xFF..FF) ⊕ (temp1) = 0
(temp1) & (0xFF..FF) ⊕ (c) & (0xFF..FF) ⊕ (temp2) = 0
(temp2) & (0xFF..FF) ⊕ (d) & (0xFF..FF) ⊕ (result) = 0
```

The optimized approach packs multiple XORs into one constraint:

```rust
// Optimized approach with same values
let builder_opt = CircuitBuilder::new();

let a = builder_opt.add_constant_64(0x1111);
let b = builder_opt.add_constant_64(0x2222);
let c = builder_opt.add_constant_64(0x4444);
let d = builder_opt.add_constant_64(0x8888);

// Optimized: single constraint for all XORs
let result = builder_opt.bxor_multi(&[a, b, c, d]);
```

This generates only one constraint:
```
(a ⊕ b ⊕ c ⊕ d) & (0xFF..FF) ⊕ (result) = 0
```

The protocol absorbs XOR operations within constraint operands at negligible cost.
